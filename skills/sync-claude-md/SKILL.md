# CLAUDE.md Sync Agent (v3)

Sync universal engineering rules from `jesselusa/claude` to all target repos. Score outcomes of recent sync PRs. On Sundays, produce a calibration report with rolling + lifetime views.

Three phases: PHASE 1 (sync), PHASE 2 (score recent outcomes), PHASE 3 (Sunday weekly report).

## PHASE 1: Sync Rules

### Target Repos

1. `jesselusa/arc`
2. `jesselusa/arc-ios`
3. `jesselusa/little-language-labs`
4. `jesselusa/pokecrm`
5. `jesselusa/palette`
6. `jesselusa/jesselusa.com`

### Workflow

1. Read the global CLAUDE.md from `jesselusa/claude`
2. For each target repo:
   a. Read that repo's CLAUDE.md
   b. Check whether branch `feat/daily-learnings-YYYY-MM-DD` already exists — if so, skip this repo
   c. Identify rules in the global CLAUDE.md not present in the project's CLAUDE.md
   d. Apply the CRITICAL FILTER (see below)
   e. For each rule that passes the filter, record your self-assessment (see Logging below)
   f. Insert qualifying rules under the matching section header. Match by semantic meaning. If no matching section exists, skip the rule.
   g. If any rules were added:
      - Create branch `feat/daily-learnings-YYYY-MM-DD`
      - Commit: `feat: sync CLAUDE.md learnings YYYY-MM-DD`
      - Push and create PR with `--assignee @me`
      - PR body lists all rules added with their sections (see format below)
      - Log the decision (see Logging below)
   h. If no rules to add, skip the repo
   i. Return to main before moving to the next repo

### CRITICAL FILTER

Only sync rules that pass ALL of the following:

**REJECT if any of these are true:**
- The rule is specific to the `jesselusa/claude` config repo itself (creating skills, syncing repos, multi-repo awareness, skill table maintenance)
- The rule is already present in the project CLAUDE.md, even if worded slightly differently
- The rule adds a single bullet to an already-detailed existing subsection — individual items appended to a list that already covers the concept don't justify a PR
- The rule is a design aesthetic opinion (color inspiration sources, typography personality, visual style guidance) and the target repo has a locked or well-defined design system (e.g., Arc's monochrome token system, PokeCRM's Pokemon themes)
- No matching section exists in the target repo's CLAUDE.md

**PREFER rules that are:**
- Direct behavioral instructions for Claude (how to ask questions, confirmation patterns, session habits, tool usage) — these are highest value regardless of size
- Entirely absent concepts, not incremental additions to existing coverage
- Applicable regardless of the project's tech stack or visual aesthetic

### PR Body Format

```
## Sync CLAUDE.md — YYYY-MM-DD

Automated PR to sync generic engineering rules from the global `jesselusa/claude` config.

### Rules added

- [Section Name] Rule text here
- [Section Name] Another rule here

---
Generated by CLAUDE.md Sync agent v3
```

### Logging

After creating PRs, switch to `jesselusa/claude` repo (main branch) and append one JSON object per PR to `evals/claude-md-sync/decisions.jsonl`:

```json
{"date":"YYYY-MM-DD","repo":"jesselusa/REPO","pr_number":123,"pr_url":"https://github.com/jesselusa/REPO/pull/123","rules_added":[{"section":"Section Name","rule":"Rule text","llm_prediction":"will_merge","llm_reasoning":"Reason here"}],"outcome":null,"outcome_date":null}
```

Commit the updated `decisions.jsonl`: `chore: log sync decisions YYYY-MM-DD` and push to main.

### Self-Assessment Fields

For each rule added, include:
- `llm_prediction`: Either `"will_merge"` (confident this belongs) or `"unsure"` (passed the filter but borderline)
- `llm_reasoning`: One-line explanation of why you included it (e.g., "Universal behavioral instruction absent from target", "Entirely new concept not covered anywhere")

These predictions are compared against the user's actual merge/close decisions during eval to calibrate the filter over time.

### Rules

- `outcome` starts as `null` — filled in during eval scoring
- One JSON object per line (JSONL format)
- Append only for new entries; eval scoring updates `outcome` / `outcome_date` fields in place on existing entries

## PHASE 2: Score Recent Outcomes (30-day window)

After Phase 1, score unresolved entries from the **last 30 days**. Older unresolved entries get auto-marked as `stale` and skipped on future runs so we don't `gh pr view` the same orphan PRs forever.

1. Read `evals/claude-md-sync/decisions.jsonl` from `jesselusa/claude`
2. For each entry where `outcome` is `null`:
   - Compute `age_days = today - entry.date`
   - If `age_days > 30`: set `outcome: "stale"`, `outcome_date: today`. Do NOT hit `gh`. These orphan PRs are presumed abandoned and are removed from the scoring loop. Log a count of how many got staled this run.
   - Else (age ≤ 30): check PR status via `gh pr view <pr_url> --json state,mergedAt,closedAt`
     - If merged: set `outcome: "merged"`, `outcome_date: mergedAt` (or today)
     - If closed (not merged): set `outcome: "closed"`, `outcome_date: closedAt` (or today)
     - If still open: leave as `null`
3. Write updated entries back to `decisions.jsonl`
4. Commit: `chore: score sync outcomes YYYY-MM-DD` and push to main
5. Log a brief summary:
   - Unresolved before: N
   - Scored this run: X merged, Y closed, Z still open
   - Staled (age > 30): S
   - Unresolved after: N - X - Y - S

**Outcome values:** `merged`, `closed`, `stale`, or `null` (still open, within window).

## PHASE 3 (Sundays only): Weekly Calibration Report

Check the day: `date +%u`. If it returns `7` (Sunday), run this phase after Phase 2.

Report two windows side by side:

1. **Lifetime** — all resolved entries (`merged` + `closed`, excludes `stale`)
2. **Rolling 30 days** — resolved entries with `outcome_date` within the last 30 days

For each window compute:
- **Total resolved PRs**
- **Merged (true positive)**: outcome = merged
- **Closed (false positive)**: outcome = closed
- **Precision**: merged / (merged + closed)
- From entries with `llm_prediction` fields:
  - **Overconfident**: predicted `will_merge` but closed
  - **Underconfident**: predicted `unsure` but merged
  - **Calibration accuracy**: correct predictions / total predictions

Also surface:
- **Stale count (lifetime)**: PRs that aged out without resolution. High stale count = you've abandoned the sync habit or the filter is too permissive.
- **Top false-positive pattern** from closed PRs (rolling 30d preferred; fall back to lifetime if fewer than 5):
  - Incremental sub-bullet addition
  - Design aesthetic on locked system
  - Already covered differently
  - LLM overconfidence
  - Other

Append to `evals/claude-md-sync/weekly-reports.md`:

```markdown
## Week of YYYY-MM-DD

### Lifetime
- Resolved PRs: X
- Precision: Y%
- Calibration accuracy: Z%
- Stale: S

### Rolling 30 days
- Resolved PRs: X'
- Precision: Y'%
- Calibration accuracy: Z'%
- Overconfident: O' | Underconfident: U'

### This week
- New PRs opened: N
- Scored this week: M
- Staled this week: T

### Top false-positive pattern (30d)
[pattern]
```

The rolling window is the primary signal. Lifetime numbers get sticky once you have 100+ entries and a great week can't move them. If the two diverge significantly (rolling precision much better than lifetime), that's evidence the filter is improving. If they're close, calibration is stable.

Commit: `chore: weekly eval report YYYY-MM-DD` and push to main.

Do NOT apply filter changes — that is done interactively via the `/eval-sync` skill.

## Important

- Do NOT rewrite or restructure any project's CLAUDE.md — only INSERT new rules
- Do NOT add project-specific rules from the global file
- Always switch back to main after finishing each repo
- If there are no new rules for any repo, skip to Phase 2
- If there are no unresolved entries and no staling to do and it's not Sunday, exit cleanly with a summary
