# CLAUDE.md Sync Agent

Sync universal engineering rules from `jesselusa/claude` to all target repos. After running, log decisions to `evals/claude-md-sync/decisions.jsonl`.

## Target Repos

1. `jesselusa/arc`
2. `jesselusa/arc-ios`
3. `jesselusa/little-language-labs`
4. `jesselusa/pokecrm`
5. `jesselusa/palette`
6. `jesselusa/jesselusa.com`

## Workflow

1. Read the global CLAUDE.md from `jesselusa/claude`
2. For each target repo:
   a. Read that repo's CLAUDE.md
   b. Check whether branch `feat/daily-learnings-YYYY-MM-DD` already exists — if so, skip this repo
   c. Identify rules in the global CLAUDE.md not present in the project's CLAUDE.md
   d. Apply the CRITICAL FILTER (see below)
   e. For each rule that passes the filter, record your self-assessment (see Logging below)
   f. Insert qualifying rules under the matching section header. Match by semantic meaning. If no matching section exists, skip the rule.
   g. If any rules were added:
      - Create branch `feat/daily-learnings-YYYY-MM-DD`
      - Commit: `feat: sync CLAUDE.md learnings YYYY-MM-DD`
      - Push and create PR with `--assignee @me`
      - PR body lists all rules added with their sections (see format below)
      - Log the decision (see Logging below)
   h. If no rules to add, skip the repo
   i. Return to main before moving to the next repo

## CRITICAL FILTER

Only sync rules that pass ALL of the following:

**REJECT if any of these are true:**
- The rule is specific to the `jesselusa/claude` config repo itself (creating skills, syncing repos, multi-repo awareness, skill table maintenance)
- The rule is already present in the project CLAUDE.md, even if worded slightly differently
- The rule adds a single bullet to an already-detailed existing subsection — individual items appended to a list that already covers the concept don't justify a PR
- The rule is a design aesthetic opinion (color inspiration sources, typography personality, visual style guidance) and the target repo has a locked or well-defined design system (e.g., Arc's monochrome token system, PokeCRM's Pokemon themes)
- No matching section exists in the target repo's CLAUDE.md

**PREFER rules that are:**
- Direct behavioral instructions for Claude (how to ask questions, confirmation patterns, session habits, tool usage) — these are highest value regardless of size
- Entirely absent concepts, not incremental additions to existing coverage
- Applicable regardless of the project's tech stack or visual aesthetic

## PR Body Format

```
## Sync CLAUDE.md — YYYY-MM-DD

Automated PR to sync generic engineering rules from the global `jesselusa/claude` config.

### Rules added

- [Section Name] Rule text here
- [Section Name] Another rule here

---
Generated by CLAUDE.md Sync agent
```

## Logging

After each run, append one entry per PR to `evals/claude-md-sync/decisions.jsonl`:

```json
{
  "date": "YYYY-MM-DD",
  "repo": "jesselusa/REPO",
  "pr_number": 123,
  "pr_url": "https://github.com/jesselusa/REPO/pull/123",
  "rules_added": [
    {
      "section": "Section Name",
      "rule": "Rule text here",
      "llm_prediction": "will_merge",
      "llm_reasoning": "One-line reasoning for why this rule belongs"
    }
  ],
  "outcome": null,
  "outcome_date": null
}
```

### Self-Assessment Fields

For each rule added, include:
- `llm_prediction`: Either `"will_merge"` (confident this belongs) or `"unsure"` (passed the filter but borderline)
- `llm_reasoning`: One-line explanation of why you included it (e.g., "Universal behavioral instruction absent from target", "Entirely new concept not covered anywhere")

These predictions are compared against the user's actual merge/close decisions during eval to calibrate the filter over time.

### Rules
- `outcome` starts as `null` — filled in during eval scoring
- One JSON object per line (JSONL format)
- Append only, never modify existing entries (eval scoring updates outcome fields in place)
